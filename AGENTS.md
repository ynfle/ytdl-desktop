## Learned User Preferences

- Prefers Bun for the desktop app when available (`bun install` / `bun run`); install Bun with `brew tap oven-sh/bun && brew install bun` if missing; npm is an acceptable alternative.
- OK with installing tools or dependencies via Homebrew when needed for setup or fixes.
- On macOS, the Electron app should fully quit when the main window is closed (not stay running in the dock with no window).
- Prefers player layout: video in the right column; play/pause, progress, and seeking span the full width below the main row (library/pages + video), not only under the video strip; while Picture-in-Picture is active, the right video column is collapsed.
- Expects the main-window transport bar (seek, play/pause, skip next) to keep working while Electron floating Picture-in-Picture is active—those actions should drive the PiP `<video>` (not only controls inside the PiP window).
- Prefers the playback **queue drawer** while playing to show **current + upcoming** only (finished tracks drop off as they play), not the full pre-play staging list; when not playing, the queue remains a staging list until play starts.

## Learned Workspace Facts

- The desktop UI lives under `desktop/` (electron-vite + React + TypeScript); it syncs with `channels.txt` / archive behavior like **`scripts/download_videos.sh`**, including ytrec, and uses the data directory as process cwd for yt-dlp. Downloaded media goes under **`videos/`** (and **`videos/rec/`** for ytrec); `channels.txt` / `downloaded.txt` stay at the data root. The library UI still understands legacy top-level uploader dirs and root **`rec/`**. For Git, prefer a source-only clone or root `.gitignore` so personal `channels.txt` and media are not committed; public repo name **ytdl-desktop**.
- `dev` and `start` must run Electron with `ELECTRON_RUN_AS_NODE` unset; if that variable is set (e.g. IDE/sandbox), `require('electron')` breaks the main process.
- Local `<video>` playback and channel logo images are served over a loopback HTTP server (video uses Range); custom file/video URL schemes from the Vite dev origin hit Chromium media URL safety checks and are not relied on for playback. On Electron, seekable “PiP” uses an always-on-top `BrowserWindow` (`floating-player.html`) fed the same loopback media URL—Document Picture-in-Picture is unreliable there; renderer logic should not treat `documentPictureInPicture.window` as “open” on Electron unless the app’s own Document PiP slot is active. In **dev**, load that page from the Vite dev server when `ELECTRON_RENDERER_URL` is set (before falling back to `loadFile` on `out/renderer/floating-player.html`) so edits to `src/renderer/floating-player.html` are not hidden behind a stale packaged copy.
- Long IPC work (e.g. channel name resolution) should return from `invoke` immediately and stream progress/results via `webContents.send`; awaiting a single long `invoke` can block the renderer from handling incremental IPC.
- Register all `ipcMain.handle` handlers before `createWindow()` so early renderer `invoke`s do not hit “No handler registered”. For floating-player (`playback:openFloatingPlayer` / close / **`playback:controlFloatingPlayer`**), use a **dedicated sync** `app.whenReady().then(() => registerFloatingPlayerIpc())` **before** the async bootstrap `whenReady().then(async …)`; set the “installed” flag **only after** `ipcMain.handle` succeeds. Also bind the same handlers on **`webContents.mainFrame.ipc`** and **`webContents.ipc`** before `loadURL` — Electron resolves renderer `invoke` on those before `ipcMain`, and `ipcMain`-only registration can still yield “No handler registered” on some Electron 39 builds. Re-bind on **`did-finish-load`** (Vite full reload / navigation can replace the frame and drop frame-level handlers).
- After preload changes, restart dev or rebuild so `out/preload` matches `src/preload` (stale preload causes missing `contextBridge` methods at runtime).
- Channel display metadata is cached on disk under Electron userData (`channel-display-cache.json`, ~7 day TTL) and is applied on load / reload list, not only after an explicit fetch.
- The Channels page can add subscriptions with **`channels:previewChannel`** (Look up) then **`channels:addChannel`**; **`desktop/src/main/channel-input.ts`** normalizes pasted YouTube URLs or bare slugs (`@handle`, `c/…`, etc.). Preview/append require a resolved row with **display name**, **loopback `logoUrl`**, and no **`row.error`**; **Sync channels** or bulk **Fetch names** blocks preview and add. **`channels:hydrateFromCache`**, **`channels:readIdentifiers`**, and the **`channels:resolveInfo`** worker use **`readChannelsLinesOrEmpty`** so a missing **`channels.txt`** yields an empty list (no **`ENOENT`** in the UI); **Sync channels** may still require the file until the first add or manual create.
- `scripts/download_videos.sh` passes `--remote-components ejs:github` on channel and ytrec `yt-dlp` paths so EJS/YouTube challenges can resolve; omitting it led to “only images” / missing format errors on ytrec.
- The library panel groups files by channel folder (`videos/<uploader>`, `videos/rec/<channel>`, or legacy uploader / `rec/` paths); each group uses a sticky header (matched channel name + avatar when available) and rows show filenames only. Users can **permanently delete** one media file via main-process **`library:deleteMedia`** (path inside data root, `LIBRARY_MEDIA_EXT`); the renderer runs **`handleLibraryFileDeleted`** to prune queue/playlist/resume and, after rescan, **`mergePositionsFromSnapshot`** only merges keys that still exist in the scan. While channel or ytrec sync runs, the main process watches `dataRoot/videos` (recursive, debounced) and emits **`sync:libraryStale`** so the renderer rescans before the job ends; **`sync:done`** still fires at the end for a final refresh (same path as Settings → Rescan library).
- Electron main process code is split under `desktop/src/main/` into focused modules (`floating-player-ipc.ts`, `register-ipc.ts`, `media-server.ts`, etc.) with a thin `index.ts` that wires `app.whenReady()`.
- Playback resume and session snapshot live in Electron userData (`playback-spot.json`, keyed by resolved data root). Resume times are stored **per library `relPath`**, not by playlist index (a new download that reuses the same path can restore an old timestamp). Renderer playback uses a **generation guard** on `<video>` loads so stale `loadedmetadata` handlers cannot apply the previous track’s resume after a quick track change. During inline play, disk writes for position are **throttled to about once per second** (with a trailing write inside the window); **immediate** flush still runs on pause, seek, tab hide, unload, stop, and floating-player close. While the **floating player** window is active, it sends throttled **`floating-player:sync`** to the main process so the main transport bar and resume saves track the PiP `<video>` (the main window’s element stays paused at handoff). The bar **forwards** seek / play-pause / toggle to that PiP `<video>` via **`playback:controlFloatingPlayer`** → **`floating-player:control`** instead of the paused main element. **Skip next** while floating uses the same **`pendingFloatingReopenRef`** handoff as end-of-track. When the main process **replaces** the floating window for the next file, it can suppress **`playback:floatingPlayerClosed`** for that programmatic teardown so the renderer does not apply a stale `currentTime`. When the floating PiP video **ends** and the playlist advances, the renderer **reopens** the floating window for the next item and skips the usual “close floating player on `currentRel` change” while that handoff is pending.
