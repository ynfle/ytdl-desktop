## Learned User Preferences

- **Bun for JS tooling (preferred):** Use Bun instead of npm/npx for this repo when possible.
  - Install deps: `bun install` (from the relevant package root, e.g. `desktop/`).
  - Run `package.json` scripts: `bun run <script>` (e.g. `bun run dev`, `bun run build`).
  - Run a package binary once (replaces `npx`): **`bunx <command>`** — e.g. `bunx tsc --noEmit`, `bunx eslint .`.
  - Install Bun if missing: `brew tap oven-sh/bun && brew install bun`.
  - npm / npx remain acceptable fallbacks if Bun is unavailable; prefer Bun when both work.
- OK with installing tools or dependencies via Homebrew when needed for setup or fixes.
- On macOS, the Electron app should fully quit when the main window is closed (not stay running in the dock with no window).
- Prefers player layout: video in the right column; play/pause, progress, and seeking span the full width below the main row (library/pages + video), not only under the video strip; while Picture-in-Picture is active, the right video column is collapsed.
- Expects the main-window transport bar (seek, play/pause, skip next) to keep working while Electron floating Picture-in-Picture is active—those actions should drive the PiP `<video>` (not only controls inside the PiP window).
- Prefers the playback **queue drawer** while playing to show **current + upcoming** only (finished tracks drop off as they play), not the full pre-play staging list; when not playing, the queue remains a staging list until play starts.
- In the queue drawer, **Queue** (user-added) section appears **above** **Up next** (library continuation); the Queue section is **hidden entirely** when empty (no header or empty-state text).
- Prefers channel/profile avatars and similar badges masked to a **circle** (or consistent round crop), not oblong or oval shapes.

## Learned Workspace Facts

- The desktop UI lives under `desktop/` (electron-vite + React + TypeScript); it syncs with `channels.txt` / archive behavior like **`scripts/download_videos.sh`**, including ytrec, and uses the data directory as process cwd for yt-dlp. Downloaded media goes under **`videos/`** (and **`videos/rec/`** for ytrec); `channels.txt` / `downloaded.txt` stay at the data root. The library UI still understands legacy top-level uploader dirs and root **`rec/`**. For Git, prefer a source-only clone or root `.gitignore` so personal `channels.txt` and media are not committed; public repo name **ytdl-desktop**. Electron **main** code is split under `desktop/src/main/` into focused modules (`floating-player-ipc.ts`, `register-ipc.ts`, `media-server.ts`, etc.) with a thin `index.ts` that wires `app.whenReady()`.
- `dev` and `start` must run Electron with `ELECTRON_RUN_AS_NODE` unset; if that variable is set (e.g. IDE/sandbox), `require('electron')` breaks the main process.
- Local `<video>` playback and channel logo images are served over a loopback HTTP server (video uses Range); custom file/video URL schemes from the Vite dev origin hit Chromium media URL safety checks and are not relied on for playback. On Electron, seekable “PiP” uses an always-on-top `BrowserWindow` (`floating-player.html`) fed the same loopback media URL—Document Picture-in-Picture is unreliable there; renderer logic should not treat `documentPictureInPicture.window` as “open” on Electron unless the app’s own Document PiP slot is active. In **dev**, load that page from the Vite dev server when `ELECTRON_RENDERER_URL` is set (before falling back to `loadFile` on `out/renderer/floating-player.html`) so edits to `src/renderer/floating-player.html` are not hidden behind a stale packaged copy.
- Long IPC work (e.g. channel name resolution) should return from `invoke` immediately and stream progress/results via `webContents.send`; awaiting a single long `invoke` can block the renderer from handling incremental IPC.
- Register all `ipcMain.handle` handlers before `createWindow()` so early renderer `invoke`s do not hit “No handler registered”. For floating-player (`playback:openFloatingPlayer` / close / **`playback:controlFloatingPlayer`**), use a **dedicated sync** `app.whenReady().then(() => registerFloatingPlayerIpc())` **before** the async bootstrap `whenReady().then(async …)`; set the “installed” flag **only after** `ipcMain.handle` succeeds. Also bind the same handlers on **`webContents.mainFrame.ipc`** and **`webContents.ipc`** before `loadURL` — Electron resolves renderer `invoke` on those before `ipcMain`, and `ipcMain`-only registration can still yield “No handler registered” on some Electron 39 builds. Re-bind on **`did-finish-load`** (Vite full reload / navigation can replace the frame and drop frame-level handlers).
- After preload changes, restart dev or rebuild so `out/preload` matches `src/preload` (stale preload causes missing `contextBridge` methods at runtime).
- Channel display metadata is cached on disk under Electron userData (`channel-display-cache.json`, ~7 day TTL) and is applied on load / reload list, not only after an explicit fetch. Channel logo / cover **.jpg** (and similar) should **not** be re-downloaded on every app launch when a valid on-disk file or cache entry already satisfies the UI.
- The Channels page can add subscriptions with **`channels:previewChannel`** (Look up) then **`channels:addChannel`**; **`desktop/src/main/channel-input.ts`** normalizes pasted YouTube URLs or bare slugs (`@handle`, `c/…`, etc.). Preview/append require a resolved row with **display name**, **loopback `logoUrl`**, and no **`row.error`**; **Sync channels** or bulk **Fetch names** blocks preview and add. **`channels:hydrateFromCache`**, **`channels:readIdentifiers`**, and the **`channels:resolveInfo`** worker use **`readChannelsLinesOrEmpty`** so a missing **`channels.txt`** yields an empty list (no **`ENOENT`** in the UI); **Sync channels** may still require the file until the first add or manual create. Rows can **remove** a line from **`channels.txt`** or **`playlists.txt`** via **`channels:removeChannel`** / **`playlists:removePlaylist`**; deletion resolves the stored line using the same **normalization** as add (equivalent URL vs handle). **`channels:removeChannel`** is blocked only while **yt-dlp media sync** (`syncRunning`) is running—not solely during channel/playlist/podcast metadata jobs.
- `scripts/download_videos.sh` passes `--remote-components ejs:github` on channel and ytrec `yt-dlp` paths so EJS/YouTube challenges can resolve; omitting it led to “only images” / missing format errors on ytrec. Both shell and `sync-jobs.ts` also pass **`--write-thumbnail --embed-thumbnail --convert-thumbnails jpg`** so new downloads get sidecar `.jpg` files and embedded cover art in MP4.
- The library panel groups files by channel folder (`videos/<uploader>`, `videos/rec/<channel>`, or legacy uploader / `rec/` paths); each group uses a sticky header (matched channel name + avatar when available) and rows show filenames only. The library scan (`library-scan.ts`) probes for sidecar thumbnails via **`library-thumbnail.ts`** (`.jpg`→`.jpeg`→`.webp`→`.png` next to each media file); **`LibraryVideo.thumbRelPath`** carries the result; the renderer resolves loopback URLs via **`useLoopbackMediaUrl`** and displays them in **`MediaThumbSlot`** (Library rows, Queue drawer, and `<video poster>`). When there is **no** sidecar or per-episode image (common for some podcasts), thumbs and player artwork should **fall back** to the channel/show **`logoUrl`**. Users can **permanently delete** one media file via main-process **`library:deleteMedia`** (path inside data root, `LIBRARY_MEDIA_EXT`); the renderer runs **`handleLibraryFileDeleted`** to prune queue/playlist/resume and, after rescan, **`mergePositionsFromSnapshot`** only merges keys that still exist in the scan. While channel or ytrec sync runs, the main process watches `dataRoot/videos` (recursive, debounced) and emits **`sync:libraryStale`** so the renderer rescans before the job ends; **`sync:done`** still fires at the end for a final refresh (same path as Settings → Rescan library).
- Podcast **catalog search** (IPC `podcasts:searchApplePodcasts`): tries Apple’s public iTunes **search** API first; if that endpoint is unavailable (e.g. HTTP **404** while **`lookup`** may still work), optional **Podcast Index** API keys—environment **`YTDL_PODCASTINDEX_KEY`** + **`YTDL_PODCASTINDEX_SECRET`** (or `PODCASTINDEX_API_KEY` / `PODCASTINDEX_API_SECRET`), or **`podcastIndexApiKey`** / **`podcastIndexApiSecret`** on app config / **`ytdl-config.json`** in userData—enable **`search/byterm`** fallback.
- Playback resume and session snapshot live in Electron userData (`playback-spot.json`, keyed by resolved data root). Resume times are stored **per library `relPath`**, not by playlist index (a new download that reuses the same path can restore an old timestamp). Renderer playback uses a **generation guard** on `<video>` loads so stale `loadedmetadata` handlers cannot apply the previous track’s resume after a quick track change. During inline play, disk writes for position are **throttled to about once per second** (with a trailing write inside the window); **immediate** flush still runs on pause, seek, tab hide, unload, stop, and floating-player close. While the **floating player** window is active, it sends throttled **`floating-player:sync`** to the main process so the main transport bar and resume saves track the PiP `<video>` (the main window’s element stays paused at handoff). The bar **forwards** seek / play-pause / toggle to that PiP `<video>` via **`playback:controlFloatingPlayer`** → **`floating-player:control`** instead of the paused main element. **Skip next** while floating uses the same **`pendingFloatingReopenRef`** handoff as end-of-track. When the main process **replaces** the floating window for the next file, it can suppress **`playback:floatingPlayerClosed`** for that programmatic teardown so the renderer does not apply a stale `currentTime`. When the floating PiP video **ends** and the playlist advances, the renderer **reopens** the floating window for the next item and skips the usual “close floating player on `currentRel` change” while that handoff is pending. The playlist uses **`explicitStartIndex`** to separate library “Up next” from user-queued tail; `PlaybackSpotSession` persists it. The load-track effect depends on a derived **`activePlaylistRel`** (`playlist[cursor]`) instead of the `playlist` array identity, preventing duplicate audio when tail-only mutations (queue add/remove, non-current delete) create a new array ref while floating PiP is playing. **Media Session** / OS controls: many surfaces (macOS Control Center, headphone **forward**) fire **`seekforward`** rather than **`nexttrack`** for “skip ahead”; the app maps **`seekforward`** to **next queue item** when the playlist has a tail, otherwise an in-file jump. **`MediaTrackNext`** / chord shortcuts and **`nexttrack`** still advance the queue; **`seekforward`** / **`seekbackward`** and absolute **`seekto`** route to the **active** player (floating PiP vs inline `<video>`).
